# ADR-005: Frontend Framework and Image Input Design

Date: 2026-06-09
Status: Accepted — **⚠ Partial**

> **Implementation status:** The React + Vite + Tailwind UI is built (`frontend/`). Drag-and-drop and click-to-pick upload are implemented. Mode A COLA stub toggle implemented (collapsible section, catalog dropdown from `GET /v1/applications`, `application=` on submit). The base64 JSON endpoint is not built — the UI uses multipart `POST /v1/check` (not `/api/v1/labels/check`). The UI is served from `frontend/dist/` via FastAPI `StaticFiles`. Mobile-optimized layout remains deferred. See [`IMPLEMENTATION_STATUS.md`](../../IMPLEMENTATION_STATUS.md).

## Context

The UI must be simple enough for non-technical users ("my mother could figure it out — she's 73"), display compliance results clearly, and support image upload. Two additional constraints shaped this decision:

1. **Image input UX:** The upload interaction must be frictionless — drag-and-drop onto a clearly marked region, with a fallback file picker. Both must be present; neither alone is sufficient.
2. **API-first image ingestion:** The prototype must leave an obvious, documented path for non-web clients to submit images programmatically (mobile apps, scanner devices, other systems). The web UI is one client of the API, not the only possible one.

## Decision

### Frontend: React + Vite + Tailwind CSS

Single-page application, built to `frontend/dist/` and served by the FastAPI backend via `StaticFiles` — single Railway service, no separate static host.

- **Vite:** fast dev server and build, minimal config
- **Tailwind:** utility-first CSS keeps layout compact and readable in Cursor
- **React:** standard choice; component model suits the upload → result workflow; no TypeScript in the initial scaffold (can be added)

### Image input UX

The upload area is a single clearly-labeled region that accepts both:
- **Drag-and-drop:** HTML5 `dragover`/`drop` events on the region
- **Click-to-pick:** clicking the region triggers a hidden `<input type="file" accept="image/*">` — no separate "Browse" button needed

The region provides visual feedback: dashed border becomes solid on drag-hover, thumbnail preview after drop/select. This is the only image input in the web UI — one target, no ambiguity.

### API image ingestion — extension point

The backend endpoint accepts images in two ways, both documented:

**Multipart form upload** (used by the web UI):
```
POST /api/v1/labels/check
Content-Type: multipart/form-data

file=<image bytes>
mode=extract
```

**JSON with base64-encoded image** (for programmatic/non-web clients):
```
POST /api/v1/labels/check
Content-Type: application/json

{
  "labels": [{ "mode": "extract", "image_b64": "<base64 string>" }]
}
```

This means a mobile app, a scanner device, or a CLI tool can submit images directly to the same endpoint without going through the browser UI. The path is documented in the README and in the OpenAPI schema (auto-generated by FastAPI).

## Consequences

- Both drag-and-drop and file-picker share the same upload handler — one code path, not two
- Non-web clients are first-class: the multipart and JSON endpoints are equivalent and both tested
- The OpenAPI schema (available at `/docs` on the deployed app) serves as the integration contract for any future client
- No mobile-optimized layout in v1 (desktop-first); the API is ready for mobile clients even if the UI is not

## Alternatives Considered

**Plain HTML + vanilla JS:** Would satisfy the UX requirements but produces harder-to-maintain code. Rejected.

**Next.js:** Server-side rendering unnecessary for this use case; adds build pipeline complexity. Rejected.

**Separate upload button + drop zone as distinct components:** Creates two visually separate targets. Rejected in favor of a single unified drop region that also responds to click — simpler for non-technical users.
